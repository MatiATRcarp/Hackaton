1. Introducción y Problema

SkillSwap nació para resolver una necesidad concreta del entorno universitario: la dificultad de los estudiantes para encontrar tutores de materias complejas dentro de su propia institución. La búsqueda de apoyo académico se realiza actualmente de manera informal —grupos de mensajería, carteleras, contactos de conocidos— sin ningún mecanismo de verificación, coordinación ni seguimiento.

El proyecto propone un marketplace de tutorías donde alumnos universitarios activos pueden registrarse como tutores, ofrecer horarios disponibles, recibir solicitudes, coordinar sesiones y obtener calificaciones verificables. Todo el flujo se registra con trazabilidad blockchain, y un sistema de recomendación basado en IA ordena los resultados de búsqueda por compatibilidad.

Problema central

  • No existe un espacio centralizado donde alumnos publiquen disponibilidad para enseñar una materia.
  • La coordinación de horarios se realiza de manera informal y sin historial.
  • No hay mecanismo de verificación de que el tutor sea un alumno activo de la carrera.
  • No existe reputación acumulada que permita al alumno evaluar la calidad antes de solicitar.

2. Solución y Funcionalidades del MVP

SkillSwap es una aplicación web que implementa un patrón MVC en PHP 8, con MySQL como motor de base de datos y módulos de IA y Blockchain integrados. La plataforma gestiona tres roles de usuario con flujos diferenciados:

RolCapacidades principales
Alumno Buscar tutores por materia, enviar solicitudes, chatear, calificar, denunciar, convertirse en tutor
Tutor Gestionar horarios, aceptar/rechazar solicitudes, chatear, finalizar tutorías (dispara pago blockchain)
Admin Aprobar/bloquear tutores, resolver denuncias, auditar transacciones blockchain, gestionar catálogo de materias

Flujo principal

  1. El tutor se registra y sube un certificado de alumno regular (pendiente de aprobación por admin).
  2. El alumno busca tutores filtrados por materia; el sistema muestra resultados ordenados por score de IA.
  3. El alumno elige un horario disponible y envía una solicitud con mensaje.
  4. El tutor acepta; un trigger SQL marca el horario como ocupado y se habilita el chat.
  5. Al finalizar, el sistema genera un tx_hash y registra el pago en la tabla blockchain.

Configuración de PDO

$pdo = new PDO("mysql:host=...", $user, $pass, [
    PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION,
    PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,
    PDO::ATTR_EMULATE_PREPARES   => false,   // prepared statements nativos
]);

La combinación de ERRMODE_EXCEPTION + EMULATE_PREPARES=false garantiza que los errores de BD se propagan correctamente y que los prepared statements son nativos del motor (no emulados), lo que previene second-order SQL injection.

5. Módulo de Inteligencia Artificial

Algoritmo de Recomendación por Score Ponderado

El sistema calcula un score de compatibilidad para cada tutor en relación al alumno que realiza la búsqueda, priorizando los resultados más relevantes:

score = (materias_coincidentes × 10)
      + (promedio_calificacion × 5)
      + (horarios_libres_activos × 2)
  • Materias coincidentes: número de materias en común entre lo que el alumno busca y lo que el tutor enseña. Es el factor con mayor peso porque es el criterio de necesidad más directo.
  • Promedio de calificación: de 1 a 5 estrellas. Afecta el ranking pero con menor peso para no excluir tutores nuevos sin calificaciones.
  • Horarios disponibles activos: tutores con mayor disponibilidad tienen una pequeña ventaja, evitando recomendar tutores con agenda llena.
Decisión arquitectónica: Se eligió score ponderado en lugar de ML porque en la etapa de MVP no hay volumen de datos suficiente para entrenar un modelo con significancia estadística. El diseño es modular: la función de score puede reemplazarse por cosine similarity o collaborative filtering cuando existan datos reales, sin modificar el resto de la aplicación.

9. Conclusión

SkillSwap integra en un único proyecto funcional cuatro de las líneas temáticas del programa de Teoría de Computación 2, más los módulos de Inteligencia Artificial y Blockchain exigidos por el examen. El resultado es un MVP con flujo de usuario completo, seguro y documentado, que demuestra dominio técnico y comprensión de las decisiones arquitectónicas tomadas en cada etapa del desarrollo.

El mayor aprendizaje del proyecto fue la integración entre capas: demostrar que la seguridad (Unidad IV), el sistema de tipos (Unidad II) y la integridad de la base de datos (Unidad III) no son características independientes sino que se refuerzan mutuamente en una arquitectura bien diseñada.

Como trabajo futuro se proyecta: reemplazo del blockchain simulado por transacciones reales en Sepolia, integración de un modelo de ML para la recomendación cuando exista volumen de datos, y extensión del sistema a redes de múltiples universidades.
